# killbug — 6轮工业级代码审查流水线

## 触发与调用

两种调用入口:

- `/killbug N` —— N 是 1-6,**直接**启动第 N 轮单轮审查(高级用户快捷入口,跳过启动器)
- `/killbug` —— **弹出启动器**,等你从下面四种意图里挑一种(回数字或直接说审查需求)。流程见下两小节

### 启动器界面(`/killbug` 无参时原样输出)

输出后**等待你回复**,不擅自进入任何流程。

```
══════════════════════════════
🐛 killbug 启动器

📋 6 轮审查覆盖范围(每轮各管一个维度,互不重叠):
  第 1 轮 · 后端           — 路由/DB/数据模型/启动入口,框架用法、SQL 安全、移植残留、契约对齐
  第 2 轮 · 前端(逻辑契约) — 接口对齐、状态管理、前端交互框架行为、模板片段契约、状态信号触发
  第 3 轮 · 样式与体验     — 视觉一致性、布局/动画/移动端、状态四态、控件四态、无障碍
  第 4 轮 · 冗余与债务     — 死代码、重复逻辑、过时垫片、未用依赖、错位注释、过期 TODO
  第 5 轮 · 安全(攻击者)   — 注入/越权/认证/CSRF/信息泄露/XSS/依赖 CVE/限流
  第 6 轮 · 收官·全局交叉  — 读所有 round-*.md(含自定义轮次),统筹审查隐式断链 + 数据流闭环 + 架构自洽,顺手接盘前几轮的跨轮交接(若有)

请选择启动方式:
  1) 全自动模式  — 1→6 连跑,每轮按方案执行
                  仅在每轮红线披露时暂停一次让你确认
  2) 单轮模式    — 指定轮次(我会再问你哪一轮 1-6)
  3) 查看进度    — 列出已完成的轮次

💬 想查上面 6 轮没覆盖的东西? 直接说,不用打数字。
   比如:
     "做一下性能审查"
     "看看接口名字、参数名乱不乱"
   我会把你说的加工成一份审查方案给你过目,你点头才开跑。

回复数字 1 / 2 / 3,或直接说要查什么
══════════════════════════════
```

> **关键约束**:每轮职责清单必须照实输出,不能省略——这是用户(包括你自己时间久了忘了、或者外部用户拿来用)做出明智选择的依据,不是装饰。

### 启动器路由(根据你回复分支)

你回复后按下面规则路由,**不擅自直接进入任何模式**:

- **回 `1`** → 进入「全自动模式」(详见下方专章),从第 1 轮 Step 1 开始
- **回 `2`** → 追问一句:"哪一轮?(1-6)",得到 N 后按单轮模式走 Step 1
  - 如果你回 `2` 时一并说了轮次(如 `2,3` 或 `2 第3轮`),直接路由到 N=3,不再追问
- **回 `3`** → 进入「查看进度」流程:
  1. 检查当前 cwd 下是否有 `_killbug-tem/` 目录
  2. 如果**没有**:输出"还未开始审查。启动第1轮 → /killbug 1,或 /killbug 选 1 全自动"
  3. 如果**有**:列出目录里的 `round-*.md` 文件(含自定义轮次),确定已完成轮次,输出:
     ```
     📊 审查进度:
     ✅ 第1轮(后端)— round-1-backend.md
     ✅ 第2轮(前端)— round-2-frontend.md
     ✅ 自定义(性能)— round-custom-performance.md
     ⏳ 下一轮:/killbug 3 (单轮) 或 /killbug 选 1 续跑全自动
     ```
- **回自然语言描述**(非 1/2/3,看起来是描述要审查什么) → 直接进入「自定义轮次」第一步加工 prompt(详见下方专章),**跳过追问**——你已经在启动器里看过提示了
  - 识别规则:回复**不是**纯数字 `1` `2` `3`,**也不是** `选1` `1模式` 这类纯路由意图,而是包含具体审查内容描述的内容(如"看哪里慢"、"查命名规范"、"扫一遍国际化"),即视为自然语言描述
  - 边界情况(无法判断意图):一句话反问"你是想查什么?能再多说两句吗?"——不擅自猜
- **回完全无关的内容**(如"你好"、"今天天气怎么样") → 一句"请回复 1 / 2 / 3,或说要查什么"再次提示,**不擅自猜测意图**

## 三种执行模式概览

| 模式 | 触发 | 行为 | 中断点 |
|------|------|------|--------|
| 单轮模式 | `/killbug N` 或启动器选 2 | 跑一轮就结束，输出三行块给下一轮命令 | Step 3 单次决策；预案外红线 |
| 全自动模式 | 启动器选 1 | 同窗口 1→2→3→4→5→6 连跑，每轮 Step 4 默认按方案 | 每轮 Step 3 红线披露时让你扫一眼批准；预案外红线；批次依赖被打破 |
| 自定义轮次 | 启动器期间直接说要查什么 | 加工 prompt → 你过目确认 → 走标准 Step 2-7 流程，报告存 `round-custom-<slug>.md` | prompt 确认；Step 3 单次决策；预案外红线 |

## 核心流程理念

**单次决策、一路到底**：你只在 Step 3 后做**一次**决策——告诉我哪些项不改，我把它们从方案里剔除，剩余按原批次时序一路执行完成，**不再因任何预披露的红线动作中途打断**。

**全自动模式下**：每轮的 Step 3 决策默认是"按方案"——你只需要看一眼红线披露汇总,确认就放行;若要剔除/强修,就在红线确认时一并说出来。

为支撑这个体验，Step 2 里"找问题"和"写方案+批次+红线"是同步完成的——你看到的是即可执行的完整包，不是半成品。

## 严格的执行流程

### Step 1：路由到对应轮次

根据传入参数 N，从下表确定该轮主题与提示词文件：

| N | 主题 | 提示词文件 | 报告文件名 |
|---|------|-----------|------------|
| 1 | 后端审查 | `prompts/round-1-backend.md` | `round-1-backend.md` |
| 2 | 前端审查 | `prompts/round-2-frontend.md` | `round-2-frontend.md` |
| 3 | 样式与体验审查 | `prompts/round-3-style.md` | `round-3-style.md` |
| 4 | 冗余与债务审查 | `prompts/round-4-debt.md` | `round-4-debt.md` |
| 5 | 安全审查 | `prompts/round-5-security.md` | `round-5-security.md` |
| 6 | 收官·全局交叉审查 | `prompts/round-6-global.md` | `round-6-global.md` |

读取对应文件作为本轮**审查指令**。

### Step 2：审查 + 同步制定修复方案（一次到位）

**开始审查前，先读 `prompts/bug-repair.md` 总纲**——把"上帝视角 / 宏观看待 / 主线 + 链路节点 / 根因解决 / 不补丁 / 本轮闭环优先 / 时序冲突规避 / 工业级方案 / 自查三问"这套理念在**找问题、写方案、做处置判断、排批次**四个动作里全程贯穿，而不是留到 Step 5 执行才用。

按提示词审查代码的同时，对每个发现的问题**同步走完 7 个动作**——这是一条逻辑链：找出问题 → 写出单点方案 → 判断跨轮影响并应用本轮闭环优先 → 决定建议处置 → 标红线 → 全方案做轮内交叉检查 → 基于交叉结论排批次时序：

1. **找问题**——每条答清：
   - **在哪**：文件:行号
   - **是什么**：BUG 表现（用户视角看到的现象 / 数据层面的偏差）
   - **为什么发生**：错误的数据流 / 调用链 / 状态切换走错了哪一步
   - **涉及哪些链路**：与上下游什么模块勾连
2. **出方案**——每条答清：
   - **改哪些文件:行号**
   - **改成什么样**：当前是 X，改成 Y（必要时附代码或 SQL 片段）
   - **为什么这么改**：相比候选方案的取舍
   - **对周边的影响**：上下游会不会被波及（既看本轮维度内的周边，也看其他轮维度的周边）
   - **修完怎么验证**：跑哪条链路 / 看哪条数据能确认修对
3. **跨轮影响判断 + 本轮闭环优先决策**——对每个方案问 3 问，应用 `bug-repair.md` 总纲的「本轮闭环优先」原则：
   - **跨维度根因?** 问题表现在本轮，根因是否在其他轮维度?（典型：前端轮发现按钮无响应，根因在后端 API 缺字段）
   - **跨维度扩散?** 修复完成后，其他轮维度的代码是否也得跟着改?（典型：改 API 路径，前端 18 处调用都要切）
   - **双向耦合?** 跟其他维度有没有双向影响?（典型：数据 NULL 边界，后端边界 + 前端兜底都要做）

   决策依据**就近抓取本轮维度内能闭环的最大集合**——能在本轮闭环就闭环（即使改动量大、表面收益低，只要本质 + 不影响周边链路）；只有以下三种情况才能不在本轮闭环（同 `prompts/bug-repair.md` L9-13；如有不一致以 bug-repair.md 为准）：
   - a. 本轮内改会跟**同轮其他批次**冲突 → 等其他批次先做
   - b. 本轮内改会跟**后续轮次**冲突 → 等后续轮次先做
   - c. **改动量极大 + 风险极高两个维度同时触发** → 整体跨轮交接

4. **建议处置**（独立必填字段，不可埋在方案正文里）——基于上一步判断，从 5 选一选：
   - **本轮完整修** —— 本轮维度内完整闭环
   - **本轮部分修 + 余下交接(归 round-N)** —— 本轮能闭环的那部分先修了，剩余部分注明交给哪轮(可以是某一固定轮，也可以是第 6 轮统筹)
   - **整体跨轮交接(理由：a 同轮冲突 / b 后续轮冲突 / c 改动多+风险大)** —— 完全不在本轮修
   - **跳过(理由：误报 / 真不需要)** —— 不是因为修不了，是因为不需要修
   - **观察** —— 暂不动，运行期复核

   ⚠️ "整体跨轮交接"是高门槛选项，不能用"收益低"做理由。本质问题 + 不影响周边 → 该修；只有 a/b/c 三种情况下才能交接。
5. **标红线**：对**本轮要修**的项（即"本轮完整修"或"本轮部分修"中本轮要做的部分）识别红线动作（删项目内文件 / 改数据库 schema / 改配置常量等），按 `bug-repair.md` 的分类标注（一类·可预披露 / 二类·真·红线）；**整体跨轮交接 / 跳过 / 观察的项不需要标红线**
6. **轮内全局交叉检查**——所有问题写完单点方案后，对**本轮要修**的方案（含部分修中本轮做的部分）做一次集合交集分析：
   - 收集所有这些方案的改动文件:行号清单
   - 找出**同文件 / 同函数 / 同链路**被多个方案命中的情况
   - 对每个交叉点显式答 4 问：**依赖**(谁是谁的前提) / **冲突**(A 改完 B 前提是否还成立) / **合并**(能否合成一次改动避免反复改同一行) / **正确顺序**(批次怎么排才不矛盾)
   - 这一步的产出**直接驱动**第 7 步排批次，作为硬约束
7. **排批次**——基于第 6 步交叉结论，把本轮要修的项排成批次时序。批次内串行、批次间按依赖。规避时序冲突的典型场景见 `bug-repair.md` 总纲。

把"找问题→出方案→处置判断→交叉分析→批次"全链条在同一次输出里给出，给你决策的是**即可执行的完整包**——你单次决策即定全局，不再因任何中间动作中途打断。

### Step 3：一次性输出完整方案（**暂停**等用户单次决策）

输出四段式结构（顺序即认知顺序：先看每个问题怎么处置、再看方案之间怎么交叉、再看批次怎么排、最后看红线）：

#### 第一段·问题清单 + 方案（按 A→D 严重程度排序）

```
【A 类·可复现缺陷】
A1. [文件:行号] 问题标题
    根因：BUG 表现 / 为什么发生 / 走错的数据流或调用链 / 涉及的上下游链路
    方案：改哪些文件:行号 / 改成什么样（必要时贴代码或 SQL）/ 为什么这么改（候选取舍）/ 对周边的影响（含跨轮）/ 修完怎么验证
    跨轮判断：跨维度根因 / 跨维度扩散 / 双向耦合 / 无（写"无"则下面建议处置就只能在「本轮完整修 / 跳过 / 观察」三选一）
    建议处置：本轮完整修 / 本轮部分修+余下交接(归 round-N) / 整体跨轮交接(理由:a 同轮冲突 / b 后续轮冲突 / c 改动多+风险大) / 跳过(理由:误报/真不需要) / 观察
    红线：触哪一类（一类·可预披露 / 二类·真·红线）/ 具体动作（如 ALTER TABLE 或删项目内 X 文件）/ 影响什么数据
          （建议处置=整体跨轮交接 / 跳过 / 观察 时本行写"无"；本轮部分修时只标本轮要做部分的红线）

A2. ...

【B 类·有依据的隐患】
B1. ...

【C 类·待确认问题】 / 【D 类·改进建议】 ...
```

> 关键约束：「建议处置」5 选**不是平等选项**——默认应是「**本轮完整修**」（详见 `bug-repair.md` 总纲「本轮闭环优先」原则）。仅当跨轮判断有实质内容、且满足 a/b/c 三种情况之一时，才允许选「本轮部分修+交接」或「整体跨轮交接」。**不能用"收益低"作不修的理由**——本质问题 + 不影响周边链路 = 该修。

#### 第二段·轮内交叉点分析（只列被多方案命中的部分，无交叉的方案不重复列）

对**本轮要修**的方案（含部分修中本轮做的部分）做集合交集分析。无交叉的方案直接进入第三段批次时序，不在本段重复。

格式样例：
```
- **`app/security.py`** 被 A1(改 fail_count 计数)+ A4(改方法签名)+ B7(加装饰器)命中
  → 正确顺序：A4 先改签名 → B7 加装饰器(套到新签名上) → A1 改逻辑(基于新签名)
  → 风险：任何顺序错位都会触发返工
- **`likes` 表插入路径** 被 A2 + B5 命中
  → 合并机会：两个方案都加唯一索引，合并为一次 ALTER TABLE
- **`sessions` schema** 被 D6(加索引) + A1、A4(读此表)命中
  → 顺序：D6 必须在批次 1 完成迁移 → A1、A4 才能跑
```

本段的依赖/合并/顺序结论是第三段排批次的**硬约束**——批次时序方案直接引用本段结论，不再重新论证。

#### 第三段·批次时序方案（基于第二段交叉结论）

格式样例：
```
批次 1：[标题]（先做的原因——例如"第二段交叉分析显示 D6 是 A1、A4 的前置"）
  - A2（独立）
  - B2（依赖 A2 的迁移逻辑）
  - D6a（独立，可并行但串行更稳）
  自查三问：达到预期？衍生 BUG？逻辑闭环？

批次 2：[标题]
  - A1
  - B5

批次 3：[标题]
  - B1, B3, B4, C1, D3, D4, D5
```

#### 第四段·红线披露汇总

格式样例：
```
本方案触及以下红线动作，一次性披露给你决策时一并批准；执行阶段不再二次确认：

- [批次 1·B3] likes 表去重 + 加唯一索引 — 触"一类·动数据·改数据库"
- [批次 1·D6a] sessions.username 加索引 — 触"一类·动数据·改数据库"
- [批次 4·D2] 删除项目根目录 verify_batch_b.py — 触"一类·动数据·删除文件"

如要单独剔除某项红线动作，请在决策时点名跳过；否则视为批准。
```

#### 末尾·决策提示

```
**请单次决策完成**。先看每条的「建议处置」5 选(本轮完整修 / 本轮部分修+交接 / 整体跨轮交接 / 跳过 / 观察)，决策语义如下：

——基础语义（二选一）——
- "按方案"      — 完全采纳方案的建议处置：本轮完整修的修，部分修的按部分修(余下交接)，整体跨轮交接的不在本轮做，跳过/观察的不动
- "全部强修本轮"— 无视交接判断，把所有问题（含整体跨轮交接、跳过、观察的）都强制在本轮修

——在基础语义上叠加微调——
- "按方案 + 强修 D2、C3"      — 把方案建议交接/跳过/观察的某几项强制改本轮修
- "按方案 + A3 也跳过"         — 在方案建议基础上把建议修的某几项额外剔除
- "全部强修本轮 + A3 跳过"     — 强修语义下单独豁免某几项

——范围/时序限定（可与上述任一组合）——
```

2. **批次依赖被剔除项打破** — 例如 A2 被剔除但 B2 依赖 A2 的迁移逻辑
3. **决策与方案建议冲突** — 例如你说"强修 D2",而 D2 方案建议跳过的理由是"修复成本极高且无业务影响"——执行前提醒一句让你确认

### 全自动模式的质量风险(写给我自己看)

同一窗口连跑 6 轮,上下文累积可能导致:
- 后期(第 4-5 轮)注意力分散,审查质量下降
- token 接近 1M 上限时自动压缩触发,关键修改细节丢失
- 前一轮的判断惯性污染下一轮(典型:第 2 轮看前端时还在用第 1 轮的后端思维)

为此**报告必须严格按精简版格式**,删一切废话——这是全自动模式可行性的关键前提。

## 自定义轮次专章

### 触发

你在启动器对话中**直接用自然语言描述要审查什么**(如"做一下性能审查"、"看看接口名字、参数名乱不乱"),或后续在任意时点说"开个自定义轮次审查 XXX"——两种入口都按本章流程处理。

> **不再有"先选 3 再追问"的二次往返**——启动器界面里已经给你看过示例,你只要直接说,就视为提交需求,直接进第一步加工 prompt。

### 流程(分三步)

**第一步:基于你描述加工出完整 prompt**

参考 `prompts/round-1-backend.md` 到 `round-5-security.md` 的标准结构,加工出包含以下要件的完整提示词:

```markdown
# 自定义轮次:<标题>

## 审查任务

请从宏观格局出发,具备全局统筹思维,打通所有链路节点,<根据用户描述展开 3-5 句话的审查任务说明>。

重点关注:

- <从用户描述提炼/延伸的关注点 1>
- <关注点 2>
- <关注点 3>
- ...(覆盖性扩展到 5-10 条)

## 输出要求

允许没发现问题。如有问题,按下列严重程度分类,并严格按 **SKILL.md Step 3 三段式**输出(第一段·问题清单+工业级方案 / 第二段·批次时序方案 / 第三段·红线披露汇总):

- **A 类·可复现缺陷**:已确认会触发的问题
- **B 类·有依据的隐患**:推断会出问题但未必触发
- **C 类·待确认问题**:线索不足需要更多信息
- **D 类·改进建议**:可改可不改

每条按 SKILL.md Step 2 第 1-5 件事的格式答清:在哪 / 是什么 / 为什么发生 / 涉及哪些链路 / 改哪些文件:行号 / 改成什么样 / 为什么这么改 / 对周边的影响 / 修完怎么验证 / 建议处置 / 红线标注。

## 修复哲学

遵守 `prompts/bug-repair.md` 总纲——上帝视角 / 根因解决 / 不补丁 / 时序冲突规避 / 工业级方案 / 自查三问。
```

**关键加工原则**:
- 用户描述里如果只提了"看性能",要延伸到"DB 查询 / N+1 / 缓存 / 大循环 / 阻塞 IO / 前端 bundle 体积"等具体面
- 用户描述里如果限定了"重点 X",优先把 X 列在重点关注的第一条
- **加工的目标是把模糊需求具体化,但不能擅自添加用户没意图的维度**(比如用户说"看性能",别擅自加"顺便也看看安全")

**第二步:展示加工后的 prompt 给你确认**

输出:
```
═══════════════════════════════════════
🛠️ 已加工 prompt(供你审阅):

<完整 prompt 全文>

═══════════════════════════════════════
确认开跑请回 "OK" 或 "开跑"。
要修订请直接说,例如:"加一条:重点关注 N+1 查询" / "去掉前端 bundle 那条" / "重点改成 DB 查询不要扯太远"。
```

你说 OK 进入第三步;你说修订指令,我**重新加工 prompt 后再次展示**(回到第二步),如此循环直到 OK。

**第三步:走标准流程**

拿加工好的 prompt 当本轮的"审查指令",**直接进入 Step 2**——审查 + 同步制定方案 + 批次时序 + 红线披露 → Step 3 单次决策 → Step 4 复述清单 → Step 5 执行(含跨轮文件冲突预扫) → Step 6 保存精简报告。

报告路径:`_killbug-tem/round-custom-<slug>.md`,slug 从用户描述自动提取(如"性能审查" → `performance`,"API 设计规范" → `api-design`,"可访问性扫描" → `a11y`)。slug 冲突加序号 `-2` `-3`。

Step 7 的过渡输出按"单轮模式"格式——三行块给下一轮命令(下一轮命令通常是 `/killbug` 重新进启动器,因为自定义轮次没有"下一个固定轮次"概念)。

### 关键约束

- **加工后的 prompt 不持久化到 `prompts/` 目录**——只在本次会话内存里活一次,跑完就忘。理由:每次自定义需求大概率不同,持久化会污染 skill 本体
- **报告保存到 `_killbug-tem/`**——参与第 6 轮收官交叉审查
- **自定义轮次不参与全自动模式的 1→6 连跑**——是独立按需触发的工具,不是流水线一部分
- **自定义轮次必须先经你确认 prompt 才开跑**——这是防 Claude 误解需求的低成本保险

## 红线协同机制（关键，贴 CLAUDE.md）

- **预披露 + 单次决策**：方案中包含的"一类·可预披露红线"（删项目内文件、改数据库 schema、改配置常量），全部在 Step 3 第三段统一披露；你 Step 4 决策时一并批准；Step 5 执行时不再二次确认
- **未披露红线必停**：执行中临时发现需要的红线动作（Step 3 没报过的），立刻停下汇报你再决定——这是 bug-repair 红线机制在新流程下的真正用武之地
- **二类·真·红线不可绕过**：扩范围（出项目目录 / 动系统 / 动宝塔）、`--force` / `--no-verify` 绕校验、提交 `.env`、`git push --force`/`reset --hard` 等危险动作——即使披露也不该出现在方案中
- 第6轮删除 `_killbug-tem/` 是你预授权动作，但**必须先屏幕通报再执行**，给救命窗口
