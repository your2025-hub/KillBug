# 第 3 轮：样式与体验审查

## 审查任务

请从宏观格局出发，具备全局统筹思维，打通所有链路节点，读透本项目的 CSS 样式、模板视觉结构、动画与过渡，找出视觉异常、布局错乱、动画卡顿、移动端适配缺失等体验问题。**本轮聚焦"视觉/体感"层——用户实际用着能不能看清、点准、滑顺；与第 2 轮（逻辑/契约）严格互斥**。

你需要从**用户实际操作的角度**去审查，而不只是看代码规范。重点关注：

- 视觉一致性：颜色、字号、间距、圆角、阴影是否成体系（含项目自有组件库的样式约定，如有）
- 布局健壮性：极长文本、空数据、超多列表项时是否会撑破或错位
- 动画/过渡：是否有抖动、跳变、卡顿，是否符合局部刷新场景的预期（如 HTMX 触发的片段替换）
- 移动端适配：触控热区、视口、横竖屏切换、虚拟键盘弹起
- **请求生命周期视觉四态**：loading/error/empty/success 视觉是否完整、清晰、合时（**信号是否触发归第 2 轮，视觉表现归本轮**——比如 loading 一闪而过让用户看不到也算缺陷）
- **控件交互四态**：hover/active/focus/disabled 是否完整
- 无障碍：键盘导航、对比度、ARIA（如项目有此要求）
- 暗色/亮色模式（如有）是否完整

**与第 2 轮的边界**：第 2 轮负责"按下按钮 JS 是否触发了请求、HTMX 契约是否对齐、数据是否同步"；本轮负责"触发后用户能不能看到、看清楚、用得舒服"。即使 JS 触发对了，但 loading 提示一闪而过用户看不到——归本轮。

## 输出要求

允许没发现问题。如有问题，按下列严重程度分类，并严格按 **SKILL.md Step 3 三段式**输出（第一段·问题清单+工业级方案 / 第二段·批次时序方案 / 第三段·红线披露汇总）：

- **A 类·可复现缺陷**：稳定可见的视觉/交互错乱
- **B 类·有依据的隐患**：边缘场景下会出现的体验问题
- **C 类·待确认问题**：疑似但需要实机验证
- **D 类·改进建议**：可优化但不紧急

每条问题必须含：文件位置（选择器 / 类名 / 行号）、根因、工业级修复方案、（如涉及）红线标注。问题清单按严重程度排序（A→D）。
