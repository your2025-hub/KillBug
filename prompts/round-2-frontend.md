# 第 2 轮：前端审查

## 审查任务

请从宏观格局出发，具备全局统筹思维，打通所有链路节点，读透本项目所有前端模板和 JS 代码，找出会导致**功能失灵、数据不同步、契约错位**的问题。**本轮聚焦"逻辑/契约"层——按下按钮能不能跑通链路、状态管不管得住、前后端接口对不对齐。视觉与体感问题严格归第 3 轮。本节示例多用 HTMX/Jinja2-fragments 体系举例,实际审查时 AI 应根据项目真实技术栈(React/Vue/原生 JS 等)对齐审查重点。**

重点关注：

- 前后端接口是否对齐（参数名、字段名、返回结构、前端交互框架与 API 路由的契约，如 HTMX 的触发模型）
- 页面状态管理是否有遗漏（局部刷新后的状态丢失、多 tab 同步、表单脏数据保留）
- **状态切换信号是否发出**：loading/error/empty/success 在请求生命周期是否正确触发 DOM/类切换（**信号视觉表现对不对、用户能不能看到——归第 3 轮**）
- 前端交互属性是否合理（如 HTMX 的 hx-target、hx-swap、hx-trigger 是否符合预期场景）
- 模板片段返回是否对齐前端 DOM 结构（如 Jinja2-fragments 的片段化返回）
- 项目自有组件库（如有）的 **JS 调用规范**（API 用法、事件契约；样式规范归第 3 轮）

**与第 3 轮的边界**：本轮只看"链路跑不跑得通、契约对不对齐、状态管不管得住"；任何视觉表现（四态长什么样、动画顺不顺、移动端能不能用、样式体系一致性）全部归第 3 轮。

## 输出要求

允许没发现问题。如有问题，按下列严重程度分类，并严格按 **SKILL.md Step 3 三段式**输出（第一段·问题清单+工业级方案 / 第二段·批次时序方案 / 第三段·红线披露汇总）：

- **A 类·可复现缺陷**：能稳定触发的功能/交互失灵
- **B 类·有依据的隐患**：特定条件触发的状态错乱
- **C 类·待确认问题**：行为异常但需更多上下文
- **D 类·改进建议**：可优化但不紧急

每条问题必须含：**文件:行号**、根因、工业级修复方案、（如涉及）红线标注。问题清单按严重程度排序（A→D）。
