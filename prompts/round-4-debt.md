# 第 4 轮：冗余与债务审查

## 审查任务

请从宏观格局出发，具备全局统筹思维，打通所有链路节点，通读本项目所有源码，找出死代码、重复逻辑、过时的兼容垫片、未使用的引入、与代码不符的注释等技术债务。**目标是让代码库瘦下来，去掉不再需要的东西。**

重点关注：

- 死代码：永远不会执行的分支、未被引用的函数/类/模块
- 重复逻辑：多处复制粘贴的相同/近似实现
- 过时的兼容垫片：移植前残留的临时桥接、polyfill、降级处理
- 未使用的 import / requirements.txt 里没用到的依赖
- 与代码不符的注释（注释说"做X"代码却做"Y"）
- 命名不一致（同一概念多种叫法）
- TODO/FIXME 标记是否还有效（很多是历史遗物）

## 输出要求

允许没发现问题。如有问题，严格按 **SKILL.md Step 3 三段式**输出（第一段·问题清单+工业级方案 / 第二段·批次时序方案 / 第三段·红线披露汇总）。本轮特殊：第一段问题清单需要按下列两个维度组织：

**维度 1：动作分类**
- **可直接删除**：删除即可，不涉及重构
- **需要重构**：有引用关系，需要先解耦再删

**维度 2：严重程度**
- **A 类·可复现缺陷**：冗余代码已经导致 BUG（如分支冲突）
- **B 类·有依据的隐患**：会误导后续开发或带来维护成本
- **C 类·待确认问题**：疑似冗余但需确认调用方
- **D 类·改进建议**：纯清洁性优化

每条问题必须含：**文件:行号**、根因（为什么判定为冗余）、工业级修复方案、（如涉及·删除项目内文件 / 改 schema 等是红线）红线标注。
